# 要件定義書

## 1. システム名称
Subtitle Studio Plus 向け  
SRT生成エンジン並立型 ローカル実行歌声ASR統合パイプライン

## 2. 背景
既存の汎用音声認識モデルは、会話音声には強い一方で、歌声に対しては十分な精度が得られないことが多い。  
特に歌唱では、母音の引き伸ばし、子音の脱落、ビブラート、リズム優先の発音変形、英語混在、表記揺れなどが生じやすく、単一モデルによる一発認識では誤認識が多発する。  
本プロジェクトは、既存の Subtitle Studio Plus を本体アプリとし、すでに搭載されている Gemini による SRT 生成経路を維持しながら、ボーカルステム化された音声を入力とし、Apple Silicon 環境上でローカル完結動作する多段パイプラインを追加することで、歌詞文字起こし精度と実用性の向上を目的とする。

## 3. 目的
本システムの目的は、Subtitle Studio Plus 上で、ボーカル分離済み音声から日本語中心の歌詞テキストを高精度に生成し、必要に応じて時刻情報付きの LRC / SRT / JSON を出力できるようにすることである。  
あわせて以下を満たすことを目指す。

1. Apple Silicon 上で高速に動作すること
2. 完全ローカル実行を基本とするローカル生成経路を追加すること
3. Swift アプリケーションから制御可能であること
4. 誤認識区間のみを再解析することで処理負荷を抑えること
5. 文字起こしだけでなく、タイムスタンプ付き字幕生成まで一貫処理できること
6. 既存の Gemini SRT 生成経路とローカル生成経路を同居させ、UI から切り替え可能であること
7. 設定画面でモデル選択と実行パラメータ調整ができること

## 6. システム概要
本システムは、Subtitle Studio Plus の Swift 本体をオーケストレータとして用い、音声チャンクを順次処理する多段構成とする。  
本プロジェクトでは、SRT 生成エンジンとして以下の二経路を並立させる。

- Gemini による既存 SRT 生成経路
- ローカル実行型歌声ASR統合パイプライン

利用者は UI で生成方式を選択し、設定画面でモデル選択および実行パラメータ調整を行う。  
ローカル生成経路の基本フローは以下の通りとする。

1. 入力音声ファイルを受け取る
2. 一定長でチャンク分割する
3. whisper.cpp + Core ML 上で Kotoba-Whisper 系モデルによりベース転写する
4. 出力テキストを評価し、疑わしい区間を抽出する
5. 疑わしい区間のみ Qwen3-ASR で再判定する
6. 統合したテキストを Forced Aligner に渡し、時刻合わせを行う
7. 辞書補正・既知歌詞照合・表記統一を実行する
8. LRC / SRT / JSON / TXT として出力する

## 7. システム構成要素

### 7.1 制御層
Subtitle Studio Plus の Swift アプリケーションが全体の制御を担う。  
役割は以下とする。

- 入力ファイル管理
- 生成方式の UI 切り替え管理
- 設定画面でのモデル選択管理
- 設定画面でのパラメータ調整管理
- チャンク生成管理
- 各外部プロセス呼び出し
- 中間結果統合
- エラー監視
- 出力ファイル生成
- 処理ログ記録

### 7.2 ベースASR層
whisper.cpp を実行基盤とし、Core ML を利用して Apple Silicon 上で高速推論を行う。  
モデルとして以下を切り替え可能とする。

- Kotoba-Whisper v2.x
- Kotoba-Whisper Bilingual

### 7.3 補強ASR層
Qwen3-ASR を用いて、ベース認識結果のうち信頼性の低い区間を再判定する。

### 7.4 アライメント層
Qwen3-ForcedAligner または同等機能を用い、歌詞断片に時刻を付与する。

### 7.5 補正層
辞書ベース補正、かな漢字変換補正、既知歌詞との照合、表記統一を行う。

### 7.6 出力層
最終成果物として以下を出力する。

- TXT
- JSON
- LRC
- SRT

## 8. 機能要件

### 8.1 入力機能

#### 8.1.1 入力ファイル形式
システムは以下の音声形式を受け付けること。

- WAV
- M4A
- FLAC
- MP3

内部処理用には WAV 16kHz mono へ正規化すること。

#### 8.1.2 入力前提条件
入力音声は、原則としてボーカルステム済みであること。  
伴奏成分が含まれていても処理自体は可能だが、精度低下を許容する前提とする。

#### 8.1.3 言語前提
主対象言語は日本語とする。  
英語混在曲も対象に含めるが、第一優先は日本語歌詞とする。

#### 8.1.4 生成方式選択
本プロジェクトでは、入力後に以下の生成方式を UI で選択可能とする。

- Gemini
- Local Pipeline

選択された生成方式に応じて Subtitle Studio Plus が処理経路を切り替えること。

### 8.2 音声前処理機能

#### 8.2.1 正規化
入力音声を内部処理向けに以下条件へ変換すること。

- mono
- 16kHz またはモデル推奨形式
- 音量正規化
- 無音検知可能な形式

#### 8.2.2 チャンク分割
音声を一定長に分割すること。  
基準値は以下とする。

- ベースASR用チャンク長: 6〜10秒
- 再判定用チャンク長: 4〜8秒
- 端数区間は前後結合または補助処理対象

#### 8.2.3 オーバーラップ
チャンク境界による歌詞切断を防ぐため、前後に 0.5〜1.5 秒程度のオーバーラップを設定可能とする。

### 8.3 ベース文字起こし機能

#### 8.3.1 推論基盤
whisper.cpp を用いてローカル推論を行うこと。

#### 8.3.2 ハードウェア最適化
Apple Silicon 上で Core ML を有効化し、可能な範囲で ANE を利用すること。

#### 8.3.3 モデル選択
以下のモデルを選択可能とする。

- Kotoba-Whisper v2.x
- Kotoba-Whisper Bilingual

#### 8.3.4 モデル選択条件
初期状態では以下のルールを採用する。

- 日本語中心曲: Kotoba-Whisper v2.x
- 英語混在率が高い曲: Kotoba-Whisper Bilingual

#### 8.3.5 推論パラメータ
設定可能項目として以下を持つこと。

- language
- temperature
- beam size
- initial prompt
- chunk length
- no speech threshold
- logprob threshold

#### 8.3.6 設定画面反映
モデル選択および推論パラメータは Subtitle Studio Plus の設定画面から変更可能とする。

### 8.4 疑わしい区間抽出機能

#### 8.4.1 目的
ベース文字起こし結果のうち、誤認識の可能性が高い区間を選別すること。

#### 8.4.2 抽出条件
以下のいずれかに該当する場合、疑わしい区間と判定する。

- 出力文字数が極端に短い
- 同一文字の不自然な連続
- ひらがな偏重で不自然
- 文として連続性がない
- 無意味な反復がある
- 信頼スコアが閾値未満
- 前後チャンクと接続不自然
- 無音でないのに空出力

#### 8.4.3 閾値設定
疑わしい判定の閾値は設定ファイルまたは設定画面で変更可能とする。

### 8.5 再判定機能

#### 8.5.1 再判定対象
疑わしい区間のみを対象とする。

#### 8.5.2 再判定モデル
Qwen3-ASR を利用すること。

#### 8.5.3 再判定単位
再判定はチャンク単位、またはフレーズ単位で実施する。

#### 8.5.4 再判定結果統合
再判定後、以下のルールで最終採用結果を決定する。

- Qwen3-ASR の結果が明らかに自然な場合は置換
- 置換で不整合が生じる場合は保留
- 保留時はログ出力のうえベース結果を維持
- 将来的に n-best 比較やスコア統合可能な構造にする

### 8.6 アライメント機能

#### 8.6.1 目的
テキスト断片に対して時間情報を付与し、字幕・歌詞同期出力を可能にする。

#### 8.6.2 機能
以下を行うこと。

- フレーズ単位時刻合わせ
- 単語単位時刻合わせ
- LRC用タイムタグ生成
- SRT用区間生成

#### 8.6.3 使用モジュール
Qwen3-ForcedAligner を第一候補とする。  
代替手段を将来的に差し替え可能な構造にする。

### 8.7 補正機能

#### 8.7.1 辞書補正
事前定義辞書に基づいて表記補正を行うこと。  
例:

- きみ → 君
- あいしてる → 愛してる

#### 8.7.2 表記統一
以下の揺れを統一対象とする。

- かな / 漢字
- 全角 / 半角
- 長音表現
- 句読点有無
- 英単語表記ゆれ

#### 8.7.3 既知歌詞照合
既知歌詞テキストが与えられた場合、候補照合による修正を可能とする。

#### 8.7.4 補正方式
補正は以下の複数段で実施可能とする。

- 単純置換
- 正規表現補正
- 編集距離ベース候補選択
- 行単位照合
- フレーズ照合

### 8.8 出力機能

#### 8.8.1 出力形式
以下を出力可能とする。

- TXT
- JSON
- LRC
- SRT

#### 8.8.2 JSON項目
JSON には最低限以下を含むこと。

- chunk id
- start time
- end time
- base transcript
- refined transcript
- final transcript
- model used
- suspicious flag
- alignment result
- correction result
- engine type

#### 8.8.3 ファイル命名
出力ファイル名は元音声ファイル名を基準に規則化すること。  
例:

- song01.final.txt
- song01.final.json
- song01.final.lrc
- song01.final.srt

### 8.9 ログ・監査機能

#### 8.9.1 ログ対象
以下をログ記録すること。

- 入力ファイル名
- 使用エンジン
- 使用モデル
- 推論開始時刻
- 推論終了時刻
- 処理時間
- エラー内容
- 再判定対象区間
- 補正実行内容
- 出力結果件数

#### 8.9.2 ログ形式
ログはテキストまたはJSON Lines形式とする。

## 9. 非機能要件

### 9.1 性能要件

#### 9.1.1 処理速度
M3 Pro 環境において、ベース転写は実時間以下または実時間近傍を目標とする。  
ただし、モデルサイズ・チャンク長・Core ML 有無で変動するため、厳密保証値ではなく目標値とする。

#### 9.1.2 再判定コスト
再判定対象は全体の 10〜30% 程度に収まることを目標とする。  
これにより、全件重再判定を避ける。

#### 9.1.3 メモリ使用量
ローカル端末上で安定動作すること。  
同時多重ロードを避け、必要に応じてモデルの逐次ロード・解放を行うこと。

### 9.2 可用性要件

- 単一ファイル処理失敗時もシステム全体が停止しないこと
- チャンク単位で再試行可能であること
- 中間成果物から再開可能であること

### 9.3 保守性要件

- モデル差し替えが容易であること
- アライナー差し替えが容易であること
- 辞書補正ルールを外部設定化すること
- Swift と Python の責務を明確に分離すること
- Gemini 経路とローカル経路の責務を明確に分離すること

### 9.4 拡張性要件
将来以下に拡張しやすい構造であること。

- 新しい歌唱ASRモデル追加
- 他言語対応
- GUI追加
- バッチ処理
- 既知歌詞との自動照合強化
- 人手修正UI
- 評価ダッシュボード追加

### 9.5 セキュリティ要件

- 原則ローカル完結
- 外部通信を必須としない
- 音声データを外部送信しない
- 一時ファイル保存先を限定する
- ログに機密情報を含めない

## 10. 外部インターフェース要件

### 10.1 Swift から whisper.cpp 呼び出し
呼び出し方式は以下のいずれかとする。

- CLI 呼び出し
- C API ブリッジ

初期実装では CLI 呼び出しを推奨する。  
理由は、実装と保守が簡単で、モデル差し替えにも強いためである。

### 10.2 Swift から Python 呼び出し
Qwen3-ASR および ForcedAligner は Python スクリプトを外部プロセスとして起動する。  
入力は以下を想定する。

- 音声ファイルパス
- 対象区間情報
- モデル設定
- 出力先パス

出力は JSON とする。

### 10.3 設定ファイル
YAML または JSON で設定可能とする。  
最低限以下を含むこと。

- モデルパス
- チャンク長
- オーバーラップ長
- language
- suspicious threshold
- output format
- correction dictionary path
- engine type

## 13. エラー処理要件
以下のケースで適切に処理継続または停止判定を行うこと。

- 入力ファイルが読めない
- 音声変換失敗
- whisper.cpp 実行失敗
- モデルファイル未検出
- Qwen3-ASR 起動失敗
- アライメント失敗
- 出力ファイル書き込み失敗
- 不正JSON返却
- 空出力多発

エラー発生時は、少なくとも以下を記録すること。

- エラー時刻
- 対象チャンク
- 実行コマンド
- 標準エラー出力
- 再試行可否

## 14. 品質要件

### 14.1 品質評価指標
システム品質は以下で評価する。

- 文字誤り率
- 単語誤り率
- 行単位一致率

- 歌詞として読める自然さ

## 15. 前提条件

- 実行環境は macOS
- Apple Silicon M3 Pro 相当
- Python 実行環境がある
- whisper.cpp 実行可能
- モデルファイル取得済み
- 入力音声はボーカルステム済み

## 16. 制約条件

- 完全な歌詞正解を保証しない

- ラップ、早口、強い加工音声では誤認識増加の可能性がある
- ハモリ混在では精度低下する
- 既知歌詞なしでは表記ゆれが残る場合がある
- アライメント精度は元転写品質に依存する
